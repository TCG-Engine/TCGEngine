# SOR — Medium-Tier Implementation Plan

Execution roadmap for the cards classified **Medium** in [sor-implement.md](sor-implement.md) (groups M0–M12 + the "Medium leaders" in C3). Card text, stats, and per-card notes live in that file — this plan only sequences the work and tells you when to invoke the skill. Follows [sor-simple-plan.md](sor-simple-plan.md), now complete (329 tests).

## How to use this plan

- **Each batch = one `swusim-implement-card` invocation.** The skill looks up card text, writes **all DSL tests first**, then **stops for review before touching implementation code**. Implement only after tests are approved.
- **Batches are ordered existing-infra → new-infra.** Phases 1–6 mostly *wire* primitives that already exist; phases 7–9 build the few genuinely-new mechanics. Do them top-to-bottom so the harder trigger work lands last.
- **Per batch:** invoke skill → review tests → approve → implement → run regression → **retro** → check off below.
- **End-of-batch retro:** after green, read `docs/swu-impl-retro.txt` and report lessons (no changes unless approved). Fold approved card-impl lessons into the `swusim-implement-card` skill itself. (memory: `feedback-batch-retro`)
- **Test runner (never host PHP):** `curl http://localhost:3400/TCGEngine/zzRegressionSWUSim.php`
- **Never commit.** The user commits manually.

> **When ready to start a batch, invoke:** `swusim-implement-card` with the listed card IDs.

## Infrastructure already in place (verified — do NOT rebuild)

The Simple tier + recent commits left most "Medium" primitives ready. Confirm with a grep before assuming any is absent:

- **`MZSplitAssign` + `ProcessSplitDamage`** (GameLogic.php) — split/divided damage. ⇒ Phase 1 is wiring, not infra.
- **`DoScry($player, $n)` + `SCRY_FINALIZE`** (GameLogic.php) — reveal top N, keep/bottom. ⇒ Phase 2.
- **`NAMECARD` decision + `NameCardUI.js`** — name-a-card. ⇒ Phase 3.
- **`OnAttackEnd` trigger dispatch** (`$onAttackEndAbilities`, `HasOnAttackEndAbility`, `AddTrigger 'OnAttackEnd'` in CombatLogic) — "when this unit completes an attack." ⇒ phases 6/7.
- **Phase-counter pattern** — `SWU_PLAYED_VILLAINY`, `SWU_ATTACKED_MANDALORIAN_{uid}`, `SWU_ENEMY_DEFEATED` set in game functions, cleared in `RegroupPhaseStart` (GameLogic.php ~879). Adding a counter is the same 2-line pattern. ⇒ Phase 4.
- **M9 cost registries** — `SWUComputePlayCost` + `$playCostModifiers` (subject-keyed) / `$playCostFieldModifiers` (source-keyed). ⇒ Phase 5.
- **Universal target handlers** — `DEAL_UNIT_DAMAGE`, `DEFEAT_UNIT`, `EXHAUST_UNIT`, `READY_UNIT`, `BOUNCE_UNIT`, `GIVE_SHIELD`, `GIVE_EXPERIENCE|N`, `HEAL_TARGET|N`, `APPLY_PHASE_BUFF|p|h`, `APPLY_PHASE_DEBUFF|p|h`, `MZMAYCHOOSE` (you-may), `MZMULTICHOOSE` (up-to-N), `OPTIONCHOOSE` (named branch).

**Genuinely NEW infra, built later in this plan:** a reactive **on-enemy-defeat trigger** (a real dispatch at defeat time, distinct from the `SWU_ENEMY_DEFEATED` phase flag) and the **`WhenDealsCombatDamage` / `WhenDefeats`** combat trigger types (Phase 7); the **`SWU_CARDS_PLAYED_COUNT`** counter (Phase 4).

## Phase 1 — Split / divided damage (`MZSplitAssign` + `ProcessSplitDamage` exist, GameLogic.php)

Done one card at a time (ordered simplest → trickiest) to de-risk the first use of `MZSplitAssign` in SOR. Pattern: collect targets → `MZSplitAssign($targets, $amount, tooltip)` (returns comma-separated `mzID:amount`) → `ProcessSplitDamage`.

- [x] **Batch 1.1a — `SOR_135`** (Palpatine unit: deal 6 damage divided among enemy units). ✅ Done (5 tests, 334 passing). New `SWUDealSplitDamage` helper — applies the whole `MZSPLITASSIGN` assignment **simultaneously** (snapshot UIDs → apply all damage → single defeat sweep), so a unit killed by its share never stales a co-target's mzID. Did NOT use the stock `ProcessSplitDamage` (deals-and-cleans-up per hit → index-shift bug). `MZSplitAssignUI.js` already included in `NextTurn.php`.
- [x] **Batch 1.1b — `SOR_092`** (Overwhelming Barrage event: give a friendly unit +2/+2 this phase, then it deals damage equal to its power divided among any number of other units). ✅ Done (4 tests, 338 passing). `SWUApplyPhaseBuff` first → read `ObjectCurrentPower` (now buffed) → split that among all units **except the dealer** (excluded by UID — "other units") → reused the new universal **`SPLIT_DAMAGE`** handler (renamed from `SOR_135`'s per-card handler). Buff stands even with no other units to damage.
- [x] **Batch 1.1c — `SOR_052`** (Redemption, Unit 6/9: heal up to 8 total across any units/bases, then deal the ACTUAL healed to itself). ✅ Done (4 tests, 343 passing). Built a new **MZSPLITASSIGN "up to" mode** (`MZSplitAssignUI.js` + param `"amount|mz1:cap1&mz2:cap2|UPTO"` — per-target caps + partial submit; backward-compatible with the damage-split format). Handler heals each via `OnHealUnit`/`OnHealBase` (clamp + heal anims), sums actual healed, self-damages Redemption. **Two engine fixes landed here (driven by the SOR_092 buff tests):** (1) **combat lethality now uses `ObjectCurrentHP`** not printed `CardHp` — +HP buffs/upgrades keep a unit alive in combat; (2) **`RegroupPhaseStart` runs `SWUCheckShrinkDefeats`** after expiring `SWUBUFF_`/`SWUDEBUFF_` — a unit kept alive by a "for this phase" +HP buff is defeated when it expires. Updated `Protector_GrantsSentinel` to the current-HP outcome. (UI "up to" mode still needs an iframe smoke test.)

**✅ Phase 1 complete** — 1.1a/b/c all done. Also fixed a **frame-animation cache race**: animations were stored as the last piece of the shared multi-piece game-state cache blob, and `SetCachePiece`/`GamestateUpdated` do *unlocked* read-modify-write of the whole blob — a concurrent stale long-poll could read the blob before the anims were added and write its copy back, clobbering the animation piece before the poll read it (symptom: split/heal animations only showed "after 2 undo retries"). Fix: frame-animations now use a **dedicated cache key** (`{gameName}_anim`, atomic overwrite, never RMW-contended) — `SetFrameAnimationCache` in `EngineActionRunner.php` + the generated `GetNextTurn.php` read + its generator emitter. Verified in the browser (animations show on the first try, every time).
